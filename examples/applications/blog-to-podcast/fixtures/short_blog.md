# 为什么 Rust 的所有权系统让并发更安全？

Rust 不像 Go / Java / Python，没有运行时垃圾回收，也没有 `synchronized` /
`Lock<T>` 这种全局锁原语作为默认机制。但奇怪的是，写并发代码反而比那些
"有 GC 有锁" 的语言更难出 data race。原因是 Rust 在编译时就用所有权 +
借用检查器把并发安全的边界画死了。

## 三条决定性规则

1. **同一时间，一份数据要么只能被多人只读，要么只能被一人读写**。这条
   不是建议，是借用检查器拒绝编译的硬规则。
2. **跨线程传递数据必须带上 `Send` 标记**。某些类型（如 `Rc<T>`）天生
   `!Send`，编译器就不让你把它丢进 `std::thread::spawn`，避免你不小心
   分享了引用计数不安全的对象。
3. **跨线程共享数据必须带上 `Sync` 标记**。`Cell<T>` 内部有可变性但不是
   `Sync`，所以编译器禁止多线程共享 `&Cell<T>`，防止隐藏的 data race。

## 实际效果

写过 Rust 的人都有这种感觉：编译过的 concurrent 程序，多数情况下行为
就是对的。不是说 Rust 让你写不出 bug，而是把 "两个线程同时读写同一个
内存" 这一大类 bug 从"运行时偶发崩溃"提前到"编译期立刻报错"。后者好
调试得多。

## 代价

代价是学习曲线。`Arc<Mutex<T>>` / `Send + Sync` / lifetime annotation
这些概念是入门后第一个真正的劝退门槛。但走过去之后，你写 Tokio
multi-task / Rayon parallel / cross-thread message passing 都会觉得
"原来并发可以这么干净"。

## 结论

Rust 的所有权不是性能优化的副产品，而是为了**让 concurrent 程序的
正确性可以被编译器证明**而设计的核心机制。这是 Rust 跟传统系统语言
最深的区别。
